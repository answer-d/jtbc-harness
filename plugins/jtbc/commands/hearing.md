# /jtbc:hearing

お客様(ユーザー)から要望を **効率的に** 引き出すヒアリングを実施します。
質問を一度に羅列せず、`requirements-interview` スキルの手順に従って
**1問ずつ・推奨案を添えて** お伺いし、決定木を依存順に解決します。

進行は **営業**(客対窓口=メインセッション)。応答は終始 `customer-relations` の丁重な敬語で行います。
(引き出した要望は裏方の課長へ引き渡し、課長サブエージェントが提案書/要件定義書を起案します。)

## 引数

- `$ARGUMENTS`(任意): 特に確認したい論点があれば指定。なければ標準の決定木に沿って進めます。

## 実行手順

1. `requirements-interview` スキルを起動する(方法論・決定木・出力例はスキルを正とする)。
2. **下調べ**: ワークスペース(既存コード/README/`.jtbc/` の既存記述)を調べ、空白を特定する。
3. `.jtbc/state.json#phase` を確認し、ヒアリングの狙いを定める:
   - `PROPOSAL` → 提案書を起こすためのヒアリング
   - `REQUIREMENTS` → 要件定義書を詰めるためのヒアリング
   - それ以外 → 「現在は <phase> です」と丁重に案内。要件追加は司令塔が変更管理(稟議)として社内で自動処理する
4. **1問ずつ**、推奨案を添えてお伺いし、お答えを待つ(自己完結で先に進めない)。
   お答えに応じて決定木の枝を展開・剪定する。
5. 重要な決定が固まるたびに「ここまでの理解」を短く確認する。
6. 提案書/要件定義書を書ける状態になったら **停止**。最終要約を提示し合意を取る。
7. 営業が合意内容を整理し、裏方の課長へ引き渡す。提案書(または要件定義書)の起案は課長が行う。メモは `.jtbc/minutes/` に残してよい。

## リソース感 (費用・期間) の伝え方

架空の「○○万円」「○○週間」は使わない。**概算トークン数**(費用)と
**概算プロンプト数=やり取りの往復回数**(期間/工数)で、オーダー(桁)を伝える。
正式な見積りは提案書(§5/§6)で提示する。詳細は `requirements-interview` スキル参照。

## このコマンドの後(お客様の操作は不要・自動で進む)

```
要望ヒアリング(/jtbc:hearing ・営業が進行)
  → 営業が合意内容を裏方の課長へ引き渡し、課長サブエージェントが提案書を起案
  → 司令塔が社内の提案審査を自動開催(根回し→ハンコ。お客様は操作しない)
  → 内部承認が得られたら、営業が提案書を承認依頼としてお客様へ自動でご提示(パス明示・ご承認待ち)
```

> 提案内容はヒアリングの場で確定・約束しない(安請け合いの禁止)。
> 社内の提案審査(自動開催)で上位承認を得てから、お客様へ正式にご提示する。
