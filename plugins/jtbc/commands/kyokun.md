# /jtbc:kyokun

教訓登録簿(lessons_learned.md)の運用コマンド。

## サブコマンド

### `add "<問題>" "<原因>" "<対策>" [横展開先]`

新しい教訓を追加します。

動作:
1. `.jtbc/lessons/lessons_learned.md` を Read (無ければテンプレから生成)
2. 次の連番 (`L-NNN`) を決定
3. 以下のフォーマットで追記:

```markdown
### L-NNN: <問題の見出し>

- 発生日: <today>
- 起票者: <現在の role>
- 状態: DRAFT

#### 発生した問題
<問題詳細>

#### 根本原因 (なぜなぜ分析)
<naze-naze スキルで「なぜ」を最低5回繰り返して得た真因>

#### 再発防止策 (仕組み > 手順 > 教育)
<対策>

#### 横展開事項
<横展開先 or "なし">

#### 関連
- 関連 REQ-ID:
- 関連 CR-ID:
- 関連 WBS-ID:
```

### `list`

教訓登録簿の一覧(L-NNN と見出しのみ)を表示。

### `export`

教訓を組織横断ナレッジ(`~/.jtbc/_org/lessons/`)へエクスポート(将来機能、MVPは TODO)。

## 出力例 (add)

```
✏️ 教訓を追加しました: L-003

タイトル: 詳細設計より実装が先行してWBS外ファイルが汚染された
状態: DRAFT
ファイル: .jtbc/lessons/lessons_learned.md

次のステップ: 課長(/jtbc:role kacho)と部長(/jtbc:role bucho)のレビューを受けてください。
PJ完了審査(司令塔が自動開催)では教訓が3件以上 APPROVED 状態である必要があります。
根本原因の掘り下げには なぜなぜ分析(naze-naze スキル)を使ってください。
インシデント由来の教訓は、障害報告書からの転記でも構いません。
```

## 完了審査との関係

JTBCではプロジェクト完了時に必ず教訓を記録します。完了審査(司令塔が自動開催)の前提条件:

- 教訓が **3件以上** ある
- すべて **APPROVED** 状態
- 各教訓に **横展開事項** が記載されている

教訓を残さないプロジェクト完了は認められません(JTBCの基本理念)。
