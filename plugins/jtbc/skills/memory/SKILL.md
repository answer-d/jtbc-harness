# 役職メモ (永続記憶 / Per-Role Memory)

JTBC のエージェント(役職)の作業記憶は **揮発的** です — サブエージェントは役目を終えると消え、
常駐 teammate もセッション終了や `/resume` で失われます。そこで、各役職は **自分の知見をディスクに
書き出して永続化** します。これにより、コールドスタートの新インスタンスでも `.jtbc/memory/` を
読み直せば文脈を再構築でき、組織としての継続性が保たれます。

> **プロジェクトの正本は別**: フェーズ・承認・成果物は `state.json` と `.jtbc/**/*.md`(正式文書)が
> 持ちます。本スキルが扱うのは、そこに載らない **役職固有の非自明な作業知** です(重複させない)。

## 置き場所

```
.jtbc/memory/
  <role>/                # 役職ごと: shacho / bucho / kacho / pmo / shunin / tantou / ses(lead は eigyo)
    INDEX.md             # 1行ポインタ集(起動時に最初に読む)
    <slug>.md            # 1ファイル1学び
  _timeline.md           # フェーズ移行の自動記録(memory_timeline フックが追記。手で触らない)
```

## 3つの動作

### 1. 起動したら、まず読む(リハイドレート)
役職として起動したら **最初に** `.jtbc/memory/<role>/INDEX.md`(あれば)を読み、過去の決定・前提・
つまずきを思い出してから作業に入る。これがコールドスタート健忘の対策の肝。

### 2. 要所で、確認なく書く(バンバン書く)
次フェーズや次回の自分に残すべきもの — 決定とその理由、前提、つまずきと対処、客の機微、設計/実装上の
制約など — を得たら、**お客様にも司令塔にも確認を取らず、即** `.jtbc/memory/<role>/<slug>.md` に記録する。
ためらわない。

> **権限は自動で通る**: `.jtbc/memory/` への書込みはプラグイン同梱の `memory_grant` フックが
> **自動承認** する。ユーザーが `settings.json` に許可を書く必要はない。バックグラウンドの
> エージェントでも確認なしで書ける。**各役職は自分の `.jtbc/memory/<role>/` のみ**書ける
> (他役職のメモへの書込みは同フックが拒否する)。

### 3. フェーズの足跡は自動で残る
`state.json#phase` が変わると `memory_timeline` フック(PostToolUse)が `.jtbc/memory/_timeline.md` に
「実時刻・役職・新フェーズ」を1行追記する。これは事実の索引なので自動。手で書かない。

## 書き方(1ファイル1学び)

```markdown
# <一行要約(これだけ読めば要点が分かる)>

<本文: 何があったか / 何を決めたか>

**なぜ重要:** <次の自分や後工程にどう効くか>
**次にどう活かす:** <再来したときの具体的アクション>
```

- 書いたら `INDEX.md` に1行ポインタを足す(`- [要約](slug.md) — フック`)。
- 既存の正式文書(提案書・要件・設計・計画)や `state.json` に **既にある内容は書かない**。
- 間違っていた・古くなったメモは更新/削除する(重複を増やさない)。

## やってはいけない
- 秘密情報・認証情報をメモに書く。
- `_timeline.md` を手で編集する(フック専用)。
- 他役職の `.jtbc/memory/<other>/` に書く(拒否される)。
- 正式文書や `state.json` の内容をそのまま転記する(重複)。
