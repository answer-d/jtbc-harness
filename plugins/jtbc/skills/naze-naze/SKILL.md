# なぜなぜ分析 (5 Whys)

ある **事象(結果)** から出発し、「なぜそれが起きたのか?」を繰り返し問うことで、
表面的な対症療法ではなく **真因(root cause)** に到達するための分析フレームワーク。

JTBC では、インシデント・不具合・社内ルール違反・スケジュール遅延などが起きたとき、
必ずこの手順で原因を掘り下げてから再発防止策を立てる。

## いつ使うか

- インシデント発生時(`incident-response` から呼ばれる)
- 障害報告書(incident_report)第5章の記入時
- 教訓登録簿(lessons_learned)の「根本原因」記入時
- テスト失敗・バグ・デグレの原因究明
- 「同じミスが繰り返される」と感じたとき

## 手順

### 1. 事象を一文で定義する (曖昧にしない)

- 悪い例: 「バグが出た」
- 良い例: 「ログイン画面で、2回連続でパスワードを誤入力するとアプリがクラッシュした」

事実のみ。憶測・評価・責任論を混ぜない。

### 2. 「なぜ?」を真因に到達するまで繰り返す

各「なぜ」は **前の答えに対して** 問う。1つの「なぜ」に複数原因があれば枝分かれさせてよい。
回数に決まりはない(「5回」は名称由来の目安)。真因に届けば3回でもよく、足りなければさらに続ける。
止めどきは下記「良い分析のための原則」に従う。

```
事象: 総合テストでデグレが発生した
 └ なぜ①: 既存機能Aのテストが落ちた
     └ なぜ②: 担当が共通関数 util.format を変更したから
         └ なぜ③: WBSタスク外の util.format を「ついでに」直したから
             └ なぜ④: 影響範囲を確認せずに着手したから
                 └ なぜ⑤: WBSに「触ってよいファイル」が明記されていなかったから
                     → 真因: タスク分解時の影響範囲定義が不十分だった
```

### 3. 真因の検証 (逆さ読みテスト)

真因から事象へ「だから(therefore)」で逆向きに読み、論理が通るか確認する。
飛躍があれば「なぜ」が足りない。

> 「触ってよいファイルが明記されていなかった。だから影響範囲を確認せず着手した。
>  だからWBS外の共通関数を変更した。だから既存テストが落ちた。だからデグレした。」 ✓

### 4. 真因に対する再発防止策を立てる

対策は **真因** に対して打つ。事象や中間の「なぜ」に対する対策は対症療法。

- 仕組みで防ぐ(最優先): hook・テンプレート・自動チェック
- 手順で防ぐ: チェックリスト・レビュー観点の追加
- 教育で防ぐ(最後の手段): 周知・注意喚起 ※人の注意力に頼る対策は弱い

上記の例なら → 「WBSテンプレートに『触ってよいファイル一覧』を必須項目化し、
未記入なら詳細設計審査を通さない(role_guard でも担当の編集対象を縛る)」。

## 良い分析のための原則

- **人ではなく仕組みを責める**: 「○○さんの不注意」で止めない。なぜ不注意が通ってしまったかを問う
- **事実ベース**: 各ステップは検証可能な事実で繋ぐ。推測には「(推定)」と明記
- **止めどき**: それ以上遡ると個人の人格や組織文化など制御不能な領域になったら一段手前で止める
- **分岐を恐れない**: 原因が複数あるなら全ての枝を掘る。主因と副因を区別する
- **対策は SMART に**: 誰が・いつまでに・何を、を必ず添える

## 出力フォーマット

```markdown
## なぜなぜ分析

- 事象: <一文>

- なぜ①: <答え>
- なぜ②: <答え>
- …(真因に到達するまで必要なだけ続ける。回数は固定しない)
- 真因: <root cause>

### 再発防止策
| # | 対策 | 種別(仕組み/手順/教育) | 担当 | 期限 |
|---|---|---|---|---|
| 1 | | | | |
```

この出力を、障害報告書・教訓登録簿・該当する稟議や課題へ転記する。
