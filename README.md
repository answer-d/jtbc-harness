# JTBC ハーネス

> AIエージェントによるソフトウェア開発に、日本の伝統的大企業のガバナンスを導入する Claude Code Plugin。
>
> 半分ジョークです。

このプラグインを使うと、あなたは **JTBC (Japanese Traditional Big Company) にシステム開発を発注したお客様** になります。
社長・部長・課長・主任・担当・外注SESからなる開発会社が、提案 → 設計 → 実装 → リリース → 完了の
工程を、稟議・フェーズゲート・会議体・インシデント対応とともに、丁重な接遇でこなします。

## なぜ作るか

伝統的日本企業を意識した意図的かつ重厚な権限分離と知識分離、プロセスへの遵守をAIエージェント開発に適用することで、AIに適切なフリクションを与え、暴走や要件逸脱の防止を狙う。
また、AIエージェントに作業を依頼したユーザーが「お客様」としての心地良い（？）体験を経験することも狙う。

## 提供する役職

| 役職    | 役割                       | できること                          | できないこと                       |
| ------- | -------------------------- | ----------------------------------- | ---------------------------------- |
| 社長    | 最終意思決定・要所のみ登場 | Go/No-Go・優先順位・重篤時の責任    | 日常運営・コード閲覧・技術選定     |
| 部長    | 承認・助言・要員払い出し   | ゲート/稟議承認・文書レビュー・増員 | 自ら手を動かす・担当へ直接指示     |
| 課長    | PM・お客様窓口             | 提案/要件/設計/計画の起案・運営判断 | コード編集・リリース単独承認       |
| 主任    | PL/テックリード            | 詳細設計・WBS・割り振り・実装       | プロジェクト承認・要件変更         |
| 担当    | 実装担当                   | 実装・テスト・SESへの割り振り       | 要件/設計/スコープ変更             |
| 外注SES | 低コスト実装支援(Haiku)    | 指示された実装・テスト              | ガバナンス文書・自律判断・会議参加 |

## 工程(フェーズ)

```
提案 →[提案審査]→《ご査収》→ 要件定義 →[PJ計画審査]→《ご査収》→
基本設計 →[基本設計審査]→《ご査収》→ 詳細設計 →[詳細設計審査]→《ご査収》→
実装 → 単体テスト → 総合テスト →[リリース判定会]→[PJ完了審査]→ 完了
```

- `[ ]` = フェーズゲート審査会=**内部承認(上位承認)**(根回し → 審査 → 承認印 🔴)。
  **お客様の操作ではなく、司令塔が自動開催** する(お客様にベンダー社内の審査を操作させない)。
- `《 》` = 客先提示(お客様確認)。**内部承認の後に自動で発火** し、課長が成果物をパス明示で
  「ご査収ください」と提示し、ご承認を賜る。**内部承認前の文書はお客様に出さない。**
- 実装〜総合テストは主任が `/jtbc:phase next` で進める。

## 主な機能

- **要望ヒアリング** — `/jtbc:hearing`(課長が1問ずつ・推奨案つきで要望を引き出す。grill-me 方式)
- **フェーズゲート(6審査会)= 内部承認** — 司令塔が **自動開催**(成果物の上位承認。客先提示の前提。お客様は操作しない)
- **客先提示(ご査収)** — 内部承認に続けて **自動発火**(内部承認済みの成果物をパス明示でお客様へ提示・ご承認。手動再提示は `/jtbc:client-review`)
- **見積りは実測値で** — 費用は概算トークン数、期間/工数は概算プロンプト数(往復回数)で表す(架空の万円・週は使わない)
- **稟議(変更管理)** — 司令塔が **自動処理**(要件/設計/技術/スコープ/工数の変更を社内で起票〜承認。お客様は操作しない。結果のみご報告)
- **会議体** — `/jtbc:meeting`(内部定例・客先定例・状況報告・役員会議・客先報告会議・上長視察)
- **インシデント対応** — `/jtbc:incident`(緊急報告 → 定期報告 → なぜなぜ分析 → 障害報告書)
- **顧客接遇** — 受注御礼・進捗報告・お詫びを丁重な敬語で
- **権限分離フック** — phase_guard / role_guard / ringi_guard が違反を物理的に阻止

## クイックスタート

### A. Marketplace 経由(将来の公式配布想定)

```bash
/plugin marketplace add answer-d/jtbc-harness
/plugin install jtbc@jtbc-harness
```

### B. ローカル開発

```bash
# プラグイン読込先に配置(例)
mkdir -p ~/.claude/plugins
ln -s "$(pwd)/plugins/jtbc" ~/.claude/plugins/jtbc
```

### 使い始め

```bash
cd /path/to/your/project
/jtbc:init                 # 依頼内容を確認 → .jtbc/ 一式 + 体制図を生成(受注御礼はまだ)
/jtbc:hearing              # 課長が1問ずつ要望を引き出す → 課長が提案書を起案
# ↓ ここから先はお客様の操作不要。弊社側で自動的に進みます
#   (自動)提案審査=内部承認(課長→部長→社長)
#   (自動)客先提示:内部承認済みの提案書を「ご査収ください」と提示 → ご承認で受注
/jtbc:status               # いつでも進捗確認
# 以下、各設計工程でも「(自動)内部承認 →(自動)客先提示」の順で 要件 → 設計 → 実装 → リリース → 完了 と続く
```

```
    ├── hooks/       ← 権限分離・統制フック (5)
    ├── templates/   ← ドキュメント雛形 (17)
    ├── modes/       ← jtbc.yaml (JTBC専用)
    └── state/       ← state schema
```
